# Zenoh Autonomous Robot Bridge Specification v1.1

## Overview
This document defines the network architecture and data contracts for the Zenoh Robot Bridge system. The platform establishes a decentralized, ultra-low latency mesh linking an Android computational edge terminal, an embedded ESP32 motor driver, and a remote laptop-based PitWall monitoring dashboard. 

The architecture isolates real-time vehicle control loops from high-bandwidth multi-stream visualization channels to ensure transmission stability over wireless links.

## 2. Data Contracts & Topic Specifications

### 2.1. `robot/drive`
* **Direction:** Edge Node $\rightarrow$ Embedded Microcontroller
* **Frequency:** 10–30 Hz (Inference-driven loop execution)
* **Payload Size:** 2 Bytes (Signed 8-bit integers)

| Byte Offset | Field Identifier | Data Type | Value Range | Functional Target Description |
|:---|:---|:---|:---|:---|
| `0x00` | `left` | `int8` | `-127` to `+127` | Left actuator velocity. Negative value triggers reverse rotation. |
| `0x01` | `right` | `int8` | `-127` to `+127` | Right actuator velocity. Negative value triggers reverse rotation. |

#### Actuator Mixing & Control Principles
The single-line tracking routine utilizes a bottom-heavy exponentially weighted path average to modulate steering corrections. Speed parameters are parsed into individual differential motor components:
* **Left Actuator Command:** $V_{\text{left}} = V_{\text{base}} - \text{turnFactor}$
* **Right Actuator Command:** $V_{\text{right}} = V_{\text{base}} + \text{turnFactor}$

#### Embedded Actuator Normalization Mapping (C Code)
```c
// Convert signed 8-bit processing targets to raw hardware timer duty cycles
int16_t left_pwm  = ((int16_t)left * 255) / 127;  
int16_t right_pwm = ((int16_t)right * 255) / 127;
Low-Level Byte Block Formats
Kotlin
byteArrayOf(127, 127)   // Nominal linear forward velocity

2.2. Camera Streams (robot/camera/*)
To protect computational loop frequencies from data blocking or serialization overhead, the video transmission topology separates individual vision tracking passes into three isolated channels.

robot/camera/
├── rgb       --> Compressed, un-annotated raw environment matrix frames
├── opencv    --> Overlaid computer vision path and tracking metrics
└── depth     --> Dense distance-estimation visualizations
Pipeline Topology Summary Table
Sub-Topic Interface	Production Rate	Payload Format	Core Objective Description
robot/camera/rgb	~15 FPS	JPEG Byte Stream	Pure external visual field data harvested directly from wide-angle lenses (Ultrawide FOV).
robot/camera/opencv	~15 FPS	JPEG Byte Stream	Target trajectory analysis frames matching internal tracking models (Cyan center references, Yellow line targets).
robot/camera/depth	1–5 FPS	Grayscale/Jet JPEG	High-density structural spatial matrix generated by local monocular quantization inference.

4. Network Discovery & Routing Topologies
4.1. Multicast Scouting Protocol (Automated Configuration)
When operating over routers supporting local broadcast advertisements, network entities auto-negotiate link targets.

UDP Multicast Target Group: 224.0.0.224:7446

Operating System Prerequisite: Android environments must explicitly claim an active platform WifiManager.MulticastLock before instantiating sessions to prevent the OS hardware from dropping incoming discovery ads.

Automated Configuration Primitives
Kotlin
// Android / Kotlin Client Mesh Initialization
val networkConfig = Config.default() 
val systemSession = Zenoh.open(networkConfig).getOrThrow()
